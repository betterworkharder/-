# 仓库内容分级与发布边界

更新日期：2026-07-17

本文件约束单仓库重构中的存储、提交和网站导出。默认规则是：研究仓库保持私有，网站只接收通过质量校验和字段白名单转换后的静态 DTO；人工审核控制 Git push，而不控制本地 JSON 生成。

## S0：可发布内容

- 已通过周报质量校验的正式条目；JSON 可以在人工复核前生成和本地预览。
- 内容导出器生成的 `issues.json`、单期 JSON 和校验清单。
- 不包含内部分数、候选池、淘汰线索、联系人、内部路径或审核过程的公开字段。

允许进入 `publish/weekly/` 和 `apps/web/src/generated/` 的只能是生成物；这些目录禁止手工编辑并由脚本校验一致性。`apps/web/src/generated/` 不提交 Git。

## S1：公司内部内容

- `config/`、`scripts/`、`templates/`、`tests/`。
- `output/weekly/*/` 中的候选池、来源审计、分析审查和过程稿。
- `state/seen-urls.jsonl` 与 `state/seen-events.jsonl`。
- `legacy/` 中用于迁移和历史追溯的材料。

S1 可以进入私有仓库，但不得直接打包到浏览器构建物。

## S2：受限内容

- `knowledge/source-extracts/` 中由内部业务材料抽取的文本。
- 未公开客户、经营数据、规划、联系人和原始敏感附件。
- `backups/` 中的工作区完整备份。

S2 默认不提交 Git。若后续确需版本化，必须先完成脱敏，并使用受控私有存储或经过审批的 Git LFS 方案。

## S3：密钥和凭据

- `.env`、API Key、访问令牌、服务账号、Cookie 和本地凭据。

S3 永不提交仓库、永不进入周报 JSON、永不注入浏览器构建。部署凭据只通过部署平台的密钥管理提供。

## 发布硬门槛

1. 目标 GitHub 仓库已确认是私有仓库。
2. 周报目录通过结构、来源和证据校验。
3. 人工已在最后一次内容变更后完成本地预览，并明确允许 Git push。
4. 导出器仅使用字段白名单，并拒绝内部路径、联系人、候选池和敏感关键词。
5. 生成数据校验值与源文件一致。
6. 前端测试和生产构建通过。

在第 1 项未确认前，不得把本工作区内容导入目标网站仓库。
